# 余钱罐

> 「余钱罐」记账小程序产品全案
>
> （基于 uni-app + uniCloud 云开发，MVP 优先落地微信小程序）

## 一、项目需求分析（立项必备）

### 1. 产品核心定位

面向有日常消费管控、分类记账与攒钱目标需求的年轻群体，以「日度限额结余滚存」为核心机制，搭配多账本管理、囤货贴纸消耗记账、心愿存款与周期挑战，提供轻量化、强反馈的个人财务收支管理工具。

### 2. 市场与背景分析

**赛道现状**

- 国内记账工具赛道成熟，头部产品（随手记、鲨鱼记账、一木记账等）覆盖了基础记账、报表统计、预算管理等通用功能，但普遍存在「功能厚重、坚持门槛高、重记录轻管控」的问题。
- 细分方向上，攒钱类、打卡类工具分散，缺少「事前限额管控 + 事中消耗记账 + 事后结余攒钱」的完整闭环产品。
- 小程序端记账工具多为简化版，普遍缺失「囤货消耗式记账」「结余多去向分配」等差异化功能，轻量化创新空间充足。

**用户痛点普遍性**

- 70%以上记账用户无法长期坚持，核心原因是记账仅为事后记录，对实际消费行为无约束感，缺少正向反馈。
- 囤货党、居家群体难以记录批量采购后的单次消耗，无法精准核算真实日度消费。
- 有攒钱目标的用户缺少「从日常消费结余里攒钱」的路径，预算结余和心愿目标割裂。

**差异化机会点**

1. **机制创新**：日限额结余可滚存、可存心愿、可入存款池，把「省钱」直接转化为「攒钱进度」，强化正向激励。
2. **场景创新**：实物贴纸+囤货消耗记账，适配批量采购、居家囤货场景，解决传统记账「一次买、多次用」的核算痛点。
3. **轻量化落地**：以小程序为载体，依托微信生态降低获客门槛，云开发模式大幅降低初期研发与运维成本。

### 3. 用户画像拆解

**画像一：职场新人月光族（核心用户）**

- **人群特征**：22-28岁，刚步入职场，月收入5k-12k，月末易超支，有攒钱意愿但自控力弱。
- **核心诉求**：控制日常消费、减少不必要开支、逐步积累存款。
- **典型场景**：每月发薪后设置单日消费限额，每天记账看剩余额度，没花完的钱攒进心愿目标（如买手机、旅行）。

**画像二：居家囤货党**

- **人群特征**：25-35岁，以宝妈、居家生活爱好者为主，习惯批量采购零食、日用品，难以核算真实日均消耗。
- **核心诉求**：精准记录囤货消耗成本、分类管理不同生活账本、直观看到每月真实花销。
- **典型场景**：超市囤货后把商品拍成贴纸录入单价，每用一件点一下自动记账，月底看分类支出统计。

**画像三：目标型攒钱用户**

- **人群特征**：学生、职场青年皆有，有明确短期消费目标（买数码、演唱会门票、奢侈品等）。
- **核心诉求**：可视化攒钱进度、从日常结余里归集资金、有约束机制避免随意支取。
- **典型场景**：创建「买笔记本电脑」心愿，每天把没花完的餐费转进去，看着进度条增长获得成就感。

**画像四：控卡轻食族（延伸用户，P2 可选模块）**

- **人群特征**：20-30岁，有减脂/控糖意识，餐饮支出占比高，已在用记账或热量 App 但两者割裂。
- **核心诉求**：一顿饭同时看清「花了多少钱」和「吃了多少热量」，减少重复录入。
- **典型场景**：晚餐外卖记账时，给每道菜贴贴纸并记热量，首页看到今日餐饮花费与「还可摄入 350 kcal / 今日缺口 +220 kcal」双指标。

## 二、产品整体架构

### 1. 产品核心价值主张

让记账从「事后被动记录」升级为「事前限额管控、事中消耗追踪、事后结余攒钱」的全链路行为，用轻量化操作和正向反馈降低记账门槛，让每一笔省下的钱都看得见、用得上、有目标。

### 2. 功能模块总览

| 一级板块 | 核心作用 |
| --- | --- |
| 首页（今日看板） | 最高频入口，展示当日限额、可用额度、快速记账入口与结余操作 |
| 账本 | 多账本管理、收支明细、分类统计、**资产账户与总余额**、贴纸库 |
| 挑战 | 日/月/年三级消费挑战、成就记录、连续打卡数据 |
| 心愿 | 心愿目标管理、通用存款池、资金存取与进度追踪 |
| 我的 | 账号信息、规则设置、数据管理（**顶栏头像进入**，非底部 Tab） |

## 三、核心功能详细设计

### 1. 日度消费限额设置

- **功能名称**：日度消费限额设置与结余结算
- **所属页面**：首页-限额卡片、我的-消费规则设置
- **功能描述**：用户可设置单日基准消费限额，系统每日自动结算当日实际消费，计算结余金额；结余可滚存叠加至后续可用额度。
- **使用场景**：用户月初/每周设定消费预算，管控日常支出；每日结束后查看结余并分配去向。
- **核心业务逻辑**：
  1. 用户设置单日基准限额（支持整数金额，最低1元），修改后次日生效，避免当日作弊调整。
  2. 当日累计支出 ≤ 基准限额时，差额为「当日结余」；超出则为「超额」。
  3. 每日按用户本地时区日切（§8.5）结算：当日累计支出 ≤ 基准限额时，差额为「当日结余」；超出则为「超额」。结余默认进入「累计结余池」，可叠加至后续可用额度；用户亦可在次日通过分配弹窗转存心愿（V1.5+ 可转存款池）。
  4. 消费抵扣顺序：优先扣减当日基准限额，再扣减累计结余池余额。
  5. 异常处理：限额设置为0时提示无效；当日已产生消费后修改限额，仅次日生效。
- **优先级**：P0

### 2. 快速手动记账

- **功能名称**：快速收支记账
- **所属页面**：首页、账本详情页
- **功能描述**：支持快速录入支出/收入，选择分类、金额、归属账本，添加备注；可为单条账目绑定普通商品贴纸（现场拍照或从素材库选用）。
- **使用场景**：消费完成后即时打开小程序记账；晚上集中补录当日消费；希望用商品图标记某笔支出（如一杯咖啡、一份外卖）以便日后回顾。
- **核心业务逻辑**：
  1. 默认选中支出类型，预置常用分类（餐饮、交通、购物等），支持自定义分类（见 §8.8）。
  2. **收入规则（见 §8.3）**：收入不计入日消费限额；可选关联收款资产账户（V1.5+）；不自动进入虚拟结余池/心愿，除非用户手动转入。
  3. 记账时可选择归属账本，默认归入总账本及最近使用的子账本。
  4. 贴纸绑定（可选，P1）：点击贴纸区域可**现场拍照**生成该条账目的商品图，或**从素材库**选择已有普通贴纸；保存后贴纸随账目展示在明细列表中。
  5. 现场拍照时可勾选「同时存入素材库」，便于下次复用。
  6. 支持补录历史账目：可选消费日期/时间，按**账目所属自然日**计入对应日的限额与挑战（见 §8.5）。
  7. 保存后实时更新当日限额卡片、对应账本数据与挑战进度。
  8. 支持编辑、删除记录，删除后数据同步回滚（含限额、挑战、虚拟资金、资产账户）。
- **优先级**：P0

### 3. 拍照识别记账

- **功能名称**：小票/截图拍照OCR记账
- **所属页面**：记账页、首页快速入口
- **功能描述**：拍摄消费小票、各大app的消费截图，自动识别金额、商户、消费时间，匹配对应分类，确认后一键入账。
- **使用场景**：超市购物、餐饮结账后有纸质小票，app消费完有账单记录，拍照/截图快速录入，避免手动输入。
- **核心业务逻辑**：
  1. 调用相机拍摄小票，上传至云端OCR接口识别/截图上传，使用ai工作流接口识别。
  2. 返回识别结果（金额、商户、日期），系统自动匹配推荐分类。
  3. 用户可手动修正识别错误的信息，确认后生成记账记录。
  4. 识别失败时直接跳转手动记账页，保留已上传图片。
- **优先级**：P1

### 4. 多账本管理

- **功能名称**：多账本创建与管理
- **所属页面**：账本页、记账页
- **功能描述**：默认存在「总账本」聚合全部数据，支持用户创建多个自定义分类账本，独立统计。**账本管「钱花在哪类场景」**，与 **资产账户**（钱在哪个渠道，见 §11）分工不同，勿混为一谈。
- **使用场景**：区分日常开销、旅行预算、装修预算等不同**消费场景**的收支，分开管理。（注：「旅行基金」等账本名指**场景分类**，与 §11.2 **投资理财**无关。）
- **核心业务逻辑**：
  1. 总账本不可删除、不可修改，自动聚合所有子账本的收支数据。
  2. 自定义账本支持新建、编辑名称与图标、删除；删除前提示「数据转移至总账本」或「彻底删除」。
  3. 每条记账记录必须归属总账本，如选择记入子账本，会同时自动计入总账本。
  4. MVP版本限制最多创建5个自定义账本，迭代版可解锁上限。
- **优先级**：P0

### 5. 商品贴纸体系（囤货贴纸 + 普通商品贴纸）

贴纸分为两类，共用「贴纸库」入口，Tab 切换管理：

#### 5.1 囤货贴纸（消耗式记账）

- **功能名称**：囤货商品贴纸创建与消耗式记账
- **所属页面**：账本-贴纸库（囤货 Tab）、记账页
- **功能描述**：拍摄商品实物生成专属囤货贴纸，记录单价与库存数量；每次消耗时点击贴纸，自动按单价生成一笔支出，库存同步扣减。
- **使用场景**：批量采购零食、日用品后，按实际使用次数记账，精准核算单日消耗成本。
- **核心业务逻辑**：
  1. 新建贴纸：上传商品图片 + 填写名称、单价、初始库存、绑定支出分类、归属账本。
  2. **首次采购记账（见 §8.4）**：创建囤货贴纸时可同步记一笔「采购支出」（总金额 = 单价 × 初始库存），计入日限额与账本；也可仅建贴纸、采购时另记，由用户选择。
  3. 消耗记账：点击贴纸，选择消耗数量（默认1件），确认后自动生成对应金额的支出记录，库存对应扣减。
  4. 库存为0时贴纸置灰，提示补货；支持手动更新库存与单价。
  5. 消耗产生的记账记录与普通记账一致，同步计入账本、限额与挑战。
- **优先级**：P1

#### 5.2 普通商品贴纸（账目装饰 / 素材复用）

- **功能名称**：普通商品贴纸与素材库
- **所属页面**：账本-贴纸库（素材 Tab）、记账编辑页
- **功能描述**：为单条账目绑定商品视觉标识，增强记账可读性与回顾体验。记账时可现场拍照，也可从素材库选择已有贴纸；素材库中的贴纸可跨账目复用，不涉及库存与单价逻辑。
- **使用场景**：点外卖、买咖啡、购小件商品后，给该笔账贴一张商品图，明细列表更直观；常用商品（如常买的牛奶、面包）提前录入素材库，后续记账一键选用。
- **核心业务逻辑**：
  1. **记账时绑定**：在记账编辑页点击贴纸区域，弹出「拍照」与「从素材库选择」两个入口；选定后贴纸预览展示在账目表单中，随记录一并保存。
  2. **素材库管理**：用户可预先新建普通贴纸（上传图片 + 填写名称，可选绑定默认分类）；支持编辑名称、更换图片、删除；按最近使用 / 创建时间排序展示。
  3. **拍照入库**：记账现场拍照时，可勾选「存入素材库」，自动以商品名称（或备注）为默认名称创建素材，供后续选用。
  4. **展示与编辑**：收支明细列表展示贴纸缩略图；进入账目详情可查看大图、更换或移除贴纸，不影响金额与分类数据。
  5. **与囤货贴纸区分**：普通贴纸不参与库存扣减、不自动生成分摊金额；囤货贴纸消耗时自动记账，普通贴纸仅作账目视觉附件。
- **优先级**：P1

> **餐次与热量（P2，可选）**：若用户开启「餐饮轻记录」，餐饮类账目可升级为餐次容器，绑定多张贴纸并记录热量，详见 §10。MVP 与 V1.5 不纳入，避免与核心记账闭环抢资源。

### 6. 日/月/年消费挑战

- **功能名称**：周期消费挑战体系
- **所属页面**：挑战页
- **功能描述**：设置日、月、年三级消费目标挑战，完成后累计连续天数与成就，挑战数据自动同步对应账本。
- **使用场景**：用户设定阶段性消费控制目标，通过打卡挑战增加记账动力与成就感。
- **核心业务逻辑**：
  1. 日挑战：当日总支出 ≤ 日限额即为挑战成功，连续成功天数累计，中断则重置。
  2. 月/年挑战：用户自定义周期总支出目标，周期内实时展示进度，结束后判定是否达标。
  3. 挑战统计口径与总账本一致，所有入账记录均计入挑战核算。
  4. 挑战成功点亮对应徽章，存入成就墙，支持生成分享图。
- **优先级**：P0（日挑战）/ P1（月/年挑战）

### 7. 心愿目标与进度追踪

- **功能名称**：心愿存款与进度可视化
- **所属页面**：心愿页、心愿详情页
- **功能描述**：创建多个心愿目标，设置目标金额；支持从当日结余、存款池转入资金，实时展示进度条。
- **使用场景**：为特定消费目标攒钱，每天把省下的钱转入心愿，直观看到攒钱进度。
- **核心业务逻辑**：
  1. 新建心愿：填写名称、目标金额、可选截止日期、上传封面图。
  2. 资金来源：当日结余手动转入、通用存款池转入、手动新增存入（仅记账虚拟资金）。
  3. 每个心愿独立展示进度条、已存金额、剩余金额、存入明细。

### 8. 通用存款池与存取规则

- **功能名称**：通用存款池与支取规则
- **所属页面**：心愿页-存款池、我的-规则设置
- **功能描述**：设立通用存款账户归集结余资金，支持存入与取出；可设置每月取出比例限制、超额消费惩罚规则。
- **使用场景**：用户归集零散结余，统一管理；通过规则约束自己随意支取存款。
- **核心业务逻辑**：
  1. 存款池资金来源：每日结余转入、心愿退回、手动存入。
  2. 取出操作：输入取出金额，资金转回「累计结余池」，可用于抵扣后续消费额度。
  3. 比例限制（P2）：可设置每月最高取出比例（如存款池总额的20%），超出则无法取出。
  4. **超额惩罚（P2，虚拟化）**：开启后，当日超额仅触发虚拟惩罚（扣减连续挑战天数、熄灭徽章、扣除虚拟积分等），**不从存款池扣减真实记账额度**；与优化建议 §2 保持一致。
  5. 所有资金均为记账虚拟额度，不涉及真实金融交易。
- **优先级**：P1（存款池基础存取）/ P2（比例限制与惩罚）

### 9. 结余多去向分配

- **功能名称**：当日结余分配
- **所属页面**：首页、每日结算弹窗
- **功能描述**：每日结算后，用户可选择将结余滚存至可用额度、存入指定心愿；**存入通用存款池为 V1.5 能力**（§8）。
- **使用场景**：每日结束时，用户对当日省下的钱进行分配，强化攒钱感知与行为闭环。
- **核心业务逻辑**：
  1. 每日按用户本地时区 **24:00** 触发日切结算（见 §8.5）；次日首次打开小程序弹出结余分配弹窗。
  2. **V1.0 选项**：① 滚存至可用额度（默认）；② 存入指定心愿；支持拆分分配。
  3. **V1.5 起**增加：③ 全部或部分存入通用存款池。
  4. 用户未手动操作时，按预设默认规则处理（默认滚存至可用额度）。
- **优先级**：P0（V1.0 含滚存 + 心愿；存款池选项随 §8 在 V1.5 开放）

### 10. 餐次记录与热量轻追踪（可选模块）

> **产品判断**：建议加入，但**不做成独立健康 App**。餐饮是年轻用户最高频支出，与贴纸体系天然契合——「每样食物一张图 + 一笔账」比单独记热量更有粘性。定位为**餐饮类记账的增强层**，用户在「我的」中手动开启，未开启时不影响原有记账流程。

- **功能名称**：餐次记录与热量轻追踪
- **所属页面**：记账编辑页（餐饮分类）、首页今日看板（双进度条）、餐次详情页
- **功能描述**：以「餐次」（早餐 / 午餐 / 晚餐 / 加餐）为容器，一顿饭对应一笔餐饮支出；可为餐内每样食物绑定贴纸并记录热量，支持整餐估总量、分项录入自动合计、或只记部分食物；保存后以组合贴纸卡片展示，并计算**热量缺口**（今日还可摄入 / 消耗减摄入），与餐饮花费并列展示。
- **使用场景**：外卖晚餐 35 元，含米饭、鸡腿、奶茶三样——分别贴贴纸、填各自热量，系统自动合计本餐 920 kcal；首页看到「今日餐饮 ¥68/¥80」与「还可摄入 350 kcal」；若开启减脂模式，另见「今日缺口 +220 kcal（消耗 1670 − 摄入 1450）」。
- **核心业务逻辑**：
  1. **开关与入口**：「我的 → 功能设置 → 开启餐饮轻记录」；仅对「餐饮」分类账目展示餐次模式，其他分类保持原样。
  2. **餐次容器**：选择餐次类型（早/午/晚/加餐）+ 填写本餐总消费金额（仍计入日限额与账本）；餐内可添加 1～N 个「食物项」。
  3. **食物项结构**：每项含名称、普通贴纸（拍照 / 素材库）、可选热量（kcal）；无贴纸也可只填名称 + 热量。
  4. **三种热量录入模式**（用户任选，可混用）：
     - **整餐模式**：只填本餐总热量，不拆分到每样食物；
     - **分项模式**：每项食物单独填热量，本餐总热量 = 各项之和；
     - **部分分项**：只给部分食物填热量，其余不填；总热量取「分项之和」与「整餐 override」中用户最终确认值。
  5. **组合展示**：明细列表中餐饮账目以「餐次卡片」呈现——左侧贴纸拼贴（最多展示 4 张，超出显示 +N），右侧金额 + 总热量；点击进入餐次详情可编辑各食物项。
  6. **日汇总与热量缺口**（与「日限额 − 已消费 = 结余」对称设计）：
     - **还可摄入（默认展示，P2）**：用户设置「日摄入目标」（如 1800 kcal），**还可摄入 = 日摄入目标 − 今日已摄入**；为负时显示「超出 X kcal」，样式对标消费超额变红。
     - **热量缺口（减脂模式可选，V2.0）**：基于 **BMR/TDEE 智能测算**（见第 8 条）得到「今日总消耗」，加上用户可选填的「当日运动额外消耗」，**热量缺口 = 总消耗 − 今日已摄入**；正值表示当日处于消耗大于摄入状态（减脂用户核心关注指标）。
     - 首页餐饮区展示：`已摄入 / 目标` 进度条 + **还可摄入** 大字；开启减脂模式时增加一行 **今日缺口 ±X kcal**（可展开查看 BMR / 活动 / 运动明细）。
     - 周视图（V2.0+）：近 7 日平均缺口、累计缺口，辅助判断趋势（不做体重曲线，避免 scope 膨胀）。
     - 热量相关指标**不参与**日消费限额计算（钱与热量两条线，避免逻辑混乱）。
  7. **BMR/TDEE 智能测算与手动覆盖**（减脂模式启用，V2.0）：
     - **自动测算（默认推荐）**：首次开启减脂模式时引导填写性别、年龄、身高、体重、日常活动强度（久坐 / 轻体力 / 中等 / 高活动）；系统用 **Mifflin-St Jeor 公式** 计算 BMR，再 × 活动系数得到 **TDEE（日总消耗估算）**；**日摄入目标**可默认等于 TDEE，或设为「TDEE − 目标缺口」（如每日希望缺 300～500 kcal 时由系统反推摄入目标）。
     - **AI 辅助（可选，非必须）**：用户可用自然语言描述生活方式（如「上班族久坐，每周健身 3 次」），AI **仅推荐活动强度档位**供用户确认，再代入公式计算——**不让 AI 直接生成 BMR/TDEE 数值**，避免幻觉与合规风险；无 AI 时用户手动选档位即可。
     - **手动覆盖（必做）**：测算结果页展示 BMR、活动系数、TDEE 三项，**任一项均可手动修改**；修改后以用户值为准，并标记「已手动调整」；支持「恢复系统测算值」一键还原。
     - **当日运动**：TDEE 为日常基准；用户可在当天额外填写「运动消耗 +X kcal」（手动或从常见运动列表选），**今日总消耗 = 当前 TDEE（或手动值）+ 当日运动**。
     - **合规提示**：测算页显著标注「仅供参考，非医疗诊断」；不根据 BMI 给出疾病或减肥处方类结论。
  8. **AI 轻建议（V2.0+，P2 后期）**：基于「今日餐饮花费进度 + 还可摄入/热量缺口 + 本餐明细」生成**参考性**文案，如「还可摄入 350 kcal，若要点夜宵建议选轻食」或「今日缺口已为负，明天可适当恢复摄入」；**明确标注非医疗/营养建议**，不做疾病相关判断；优先用规则模板（如还可摄入 < 200 kcal、缺口转负时提示），大模型后置。
- **不做范围**（控制 scope）：
  - 不做完整营养库（蛋白质/碳水/脂肪等宏量营养素）
  - 不做食物识别自动算热量（可作为更远期探索）
  - 不让 AI 独立输出 BMR/TDEE 数字（仅辅助选活动档位，数值由公式计算）
  - 不替代 Keep、薄荷健康等专业工具
- **优先级**：P2（基础餐次 + 热量，V1.5 后迭代）/ AI 建议 V2.0+

### 11. 资产账户与余额管理

> **产品判断**：**建议加入**。很多用户的核心诉求是「各渠道还有多少钱」，不一定愿意逐笔记账。与现有「账本（场景分类）」并列增设 **资产账户（资金渠道）**：记账时可选扣减哪个账户；懒人可直接改余额数字；首页/账本页展示 **总余额**。

> **概念区分**：| 维度 | 账本 | 资产账户 |
> | --- | --- | --- |
> | 回答的问题 | 这笔钱**花在哪类**（餐饮、旅行） | 钱**从哪个渠道**出（微信、工行卡） |
> | 典型用途 | 分类统计、挑战、限额 | 余额查看、总资产汇总 |
> | 虚拟/真实 | 多为统计维度 | 对应真实或专项资金池 |

- **功能名称**：资产账户与余额管理
- **所属页面**：账本页-资产账户区、账户详情页、记账页-付款账户选择、首页总余额卡片（可选展示）
- **功能描述**：用户可创建多个资产账户（如微信零钱、支付宝、工行储蓄卡、现金）；每笔收支可选关联账户并自动增减其余额；支持**直接修改余额**（调账）而无需补录流水；可勾选参与汇总的账户，展示**总余额**。
- **使用场景**：
  - 精细型：每笔记账选「从微信扣」，余额随流水自动变。
  - 懒人型：月底打开工行卡，直接把余额改成 3200，不补历史明细。
  - 总览型：勾选微信 + 支付宝 + 工行，首页显示「可支配总余额 ¥8,650」。
- **核心业务逻辑**：
  1. **预置账户类型**：日常账户（微信、支付宝、银行卡、现金）——**默认计入总余额、默认关联日限额消费**。
  2. **两种维护方式**（可并存）：
     - **流水驱动**：记账时选择「付款/收款账户」，支出扣减、收入增加，余额 = 初始余额 + 累计流水；
     - **直接改数**：账户详情点「调整余额」，输入目标金额，系统生成一条「余额调整」记录（可选填备注），差额计入流水以便追溯。
  3. **初始余额**：新建账户时设置期初余额；后续以流水 + 调账为准。
  4. **总余额（可支配）**：`总余额 = Σ 用户勾选「计入可支配总览」的日常账户当前余额`；首页/账本顶栏展示，点击进账户列表。
  5. **与日限额关系**：默认仅 **日常账户** 的**消费类支出**计入日消费限额；公积金专户支出、投资买卖（资产形态转换）均**不计入**日限额。
  6. **转账**：账户间转账（如微信 → 银行卡）记两笔或一笔转账流水，总额不变、各账户此消彼长。
- **优先级**：P1（基础账户 + 改余额 + 总余额，V1.5）

#### 11.1 公积金等专项账户（可选，P2）

> **与投资理财无关**。公积金、医保个人账户等**政策限定、不可随意支配**的资金，单独归类，不要与 §11.2 的基金/股票/理财混淆。

- **账户类型**：专项账户（公积金、医保个人账户等），创建时选类型模板。
- **默认规则**：
  - **默认不计入**「可支配总余额」与 **日消费限额**（避免虚增日常购买力）；
  - 用户可在设置里手动勾选「计入总览」，但不建议默认开启。
- **记账**：支出可选「从公积金扣」，仅做记录与专项账户余额扣减，**不影响**日限额卡片。
- **余额维护**：支持流水记账 + 直接改数（与日常账户相同）；可设「年度提取额度」参考值（用户手填，系统只做剩余展示，不对接政府接口）。
- **优先级**：P2（V1.5 后）

#### 11.2 投资理财账户：基金 / 股票 / 理财等（可选，V2.0）

> **即用户所指的「基金」——理财型资产**（公募基金、股票、银行理财、债券、黄金等），**不是公积金**。可以记，但应与日常零钱、日限额**完全隔离**；只做**自记参考**，不做真实交易与投顾。

- **功能名称**：投资理财账户与持仓管理
- **所属页面**：账本页-投资 Tab、持仓详情页、账户转账/买卖记录页
- **功能描述**：用户可创建投资账户（如「支付宝基金」「同花顺股票」），其下维护多只**持仓**（如某指数基金、某只股票）；记录当前市值或份额，支持买入/卖出/分红等流水，或直接改市值；账本页独立展示「投资资产合计」，可与可支配余额合并查看「总资产」。
- **使用场景**：
  - 懒人型：每月打开一次，把「支付宝基金」总市值改成 5.2 万，不记每笔申购。
  - 精细型：记录「从工行卡买入基金 3000」「卖出股票回款 8000」，现金账户与投资账户联动变动。
  - 总览型：查看「可支配 ¥8,650 + 投资 ¥52,000 = 总资产 ¥60,650」。
- **核心业务逻辑**：
  1. **账户 + 持仓两级**：投资账户（渠道）→ 持仓列表（单只基金/股票/理财产品，名称 + 可选代码）。
  2. **两种维护方式**（与日常账户一致）：
     - **直接改数**：更新某持仓当前市值（或份额 × 单价），生成「市值调整」记录；
     - **流水驱动**：记录「买入 / 卖出 / 分红 / 定投」——**买入**从指定日常账户扣款、增加持仓成本；**卖出**减少持仓、回款至日常账户；均为**资产形态转换**，**不计入**日消费限额与餐饮等消费统计。
  3. **投资资产合计**：`投资合计 = Σ 各持仓当前市值`；默认**不计入**可支配总余额。
  4. **总资产（可选展示）**：`总资产 = 可支配总余额 + 投资资产合计 + 用户勾选的专项账户`；首页/账本可切换「只看可支配 / 含投资 / 全资产」视图。
  5. **简易盈亏（可选）**：用户手填「持仓成本」，系统展示浮动盈亏 = 市值 − 成本，**不提供**投资建议或预测。
  6. **与心愿/存款池**：投资资产**不自动同步**至虚拟存款池/心愿，避免虚拟资金与真实自记资产混淆。
- **不做范围**：
  - 不对接行情 API、不自动拉取净值/股价
  - 不支持真实申购、赎回、下单
  - 不提供投顾、荐股、理财销售
  - 产品内标注「投资数据为用户自记，仅供参考」
- **优先级**：P2 / V2.0

#### 11.3 不做范围（资产模块通用）

- 不对接银行/微信/支付宝 API 自动同步真实余额（可远期探索，MVP 不做）
- 资产账户余额为**用户自记参考**，非真实资金托管
- 不提供实时行情、官方公积金查询、证券交易能力

### 12. 余钱罐主视觉与罐体皮肤体系

> **产品判断**：首页「透明玻璃余钱罐 + 罐内钱币堆叠」是品牌核心记忆点（见 §四.1）；**罐体皮肤**作为兴奋型需求，强化个性化与分享传播，但不影响限额计算与「还可花」可读性。V1.0 仅默认罐；皮肤体系 V1.5 小范围验证，V2.0 正式扩展。

#### 12.1 默认主视觉（V1.0 · 必做）

- **造型**：半透明磨砂玻璃储蓄罐，圆角罐身，内高光 + 柔阴影。
- **罐内**：stylized 硬币 / 纸币叠层（**非写实人民币照片**）；配色 `#C2F2C8 → #89E59C → #4FD974 → #25CC5D`，底层密、往上渐疏。
- **高度映射**：`堆叠高度% = 还可花 ÷ 当日可用 × 100%`（与 §8.1 公式一致）。
- **数字绑定**：罐旁大号「还可花 ¥XX」为锚点；辅助「已花 / 限额」小字。
- **交互**：点击整张限额卡 → 限额设置页；记一笔支出后罐内钱币**平滑下沉**（见 §12.4）。
- **超额态**：钱币见底 + 浅红提示 `#F53F3F on #FFF9F9`，逻辑不变。

#### 12.2 皮肤体系分层（架构 · 避免返工）

| 层级 | 内容 | 是否随皮肤变化 |
| --- | --- | --- |
| **基础层** | 还可花金额、堆叠高度算法、超额/结余业务逻辑 | **不变** |
| **皮肤层** | 罐身材质、罐口装饰、硬币样式、罐周光晕/小挂件 | **可换** |
| **动效层** | 支出飞出、日结落入、滚存垫高等动画样式 | 可换（可选） |

**皮肤不可改动**：堆叠高度与金额的对应关系、锚点数字色 `#25CC5D`、超额红色语义、合规 disclaimer。

#### 12.3 皮肤品类与版本节奏

#### 12.4 动画规范（与皮肤解耦）

| 场景 | 动效 | 时长 | 版本 |
| --- | --- | --- | --- |
| 记一笔支出 | 罐顶 1–2 枚硬币轻飘出 → 淡出；堆叠平滑下沉至新高度 | 400–600ms | V1.0 可静态；V1.1+ 动画 |

## 五、非功能需求补充

### 1. 性能要求

- 小程序首屏加载时间 ≤ 2s，核心操作（记账保存、页面切换）响应时间 ≤ 500ms。
- 千条以内记账明细列表滚动无卡顿，图片加载支持懒加载与压缩。
- 云函数平均响应时间 ≤ 300ms，峰值并发下无明显延迟。

### 2. 多端适配要求

- MVP 优先适配微信小程序，兼容 iOS / 安卓端微信最新版本。
- 适配主流手机屏幕尺寸（320px-430px 宽度），全面屏底部安全区适配。
- 基于 uni-app 框架保留向支付宝小程序、原生 App 扩展的能力，核心代码复用率 ≥ 80%。

### 3. 安全合规要求

- 严格遵循《个人信息保护法》《网络安全法》，仅收集必要的用户信息（微信昵称、头像、openid）。
- 用户财务数据按 openid 云端隔离，仅本人可访问；数据传输全程加密。
- 明确产品为「记账工具」，不涉及真实资金交易、金融理财、信贷等业务，规避金融合规风险。
- 拍照功能需明确告知用途，获取用户相机授权后方可调用。
- 提供**用户协议**、**隐私政策**页面；设置内提供**账号注销**与**个人数据删除**入口（《个保法》合规）。
- 虚拟记账、投资自记、热量参考等场景均需在界面显著位置标注免责声明。

### 4. 账号体系要求

- 采用微信一键登录，无需额外注册，openid 作为用户唯一标识。
- 支持游客模式（本地存储数据），登录后自动同步云端数据；冲突策略见 §8.7。
- 支持手动触发数据备份，云端保留最近30天数据快照。

### 5. 数据统计要求

- 接入微信小程序官方统计，监控核心指标：日活、新增用户、次日留存、7日留存。
- 埋点核心事件：记账次数、心愿创建数、挑战完成数、贴纸使用次数、功能点击率。
- 支持按日/周查看用户活跃与功能使用情况，辅助产品迭代决策。

### 6. 消息通知（V1.0 基础 / V1.5 完善）

- 接入微信**订阅消息**，在用户授权后发送：当日超额提醒、日结结余待分配、连续挑战即将中断提醒。
- 授权时机：首次设置限额后、首次开启挑战后，说明用途再请求授权；支持在「我的-通知设置」关闭。
- 频控：同一类型消息每日最多 1 条，避免打扰。

## 八、关键规则补充（开发前必读）

本节统一前文分散的业务规则，消除 MVP 边界与资金流转歧义。

### 8.1 虚拟资金流状态机

产品内存在五类**虚拟记账额度**（非真实资金）：

| 池/概念 | 含义 | 典型变化 |
| --- | --- | --- |
| **日基准限额** | 用户设定的单日消费上限 | 我的-限额设置，修改次日生效 |
| **累计结余池** | 历史未分配结余的滚存池 | 日结盈余默认进入；可从心愿/存款池退回 |
| **当日可用额度** | 今天实际能花的额度 | 见下方公式 |
| **心愿账户** | 各心愿独立的虚拟已存金额 | 结余分配转入、手动虚拟存入、取出退回 |
| **通用存款池** | 归集结余的虚拟账户（V1.5+） | 结余转入、心愿退回、取出至结余池 |

**当日可用额度（消费前）**：

```
当日可用 = 日基准限额 + 累计结余池可抵扣额 − 当日已消费
```

**消费抵扣顺序**：先扣减「日基准限额」对应部分，再扣减「累计结余池」。

**当日结余**：

```
当日结余 = max(0, 日基准限额 − 当日基准限额内已消费)
```

（超额日结余为 0；超额部分仅触发提醒/虚拟惩罚，不生成正向结余。）

**日结分配流向（V1.0）**：

```
当日结余 ──→ 滚存至累计结余池（默认，叠加后续可用额度）
         ──→ 存入指定心愿（扣减可分配结余，增加心愿已存）
```

**V1.5 增加**：`当日结余 ──→ 通用存款池`

**心愿 ↔ 结余池**：心愿取出默认退回累计结余池；不可直接变为「已消费」。

**统一入口**：首页 / 心愿页提供「结余与分配」管理页，查看累计结余池余额及历史分配记录。

### 8.2 虚拟资金 vs 真实资产账户

| | 虚拟资金（结余池/心愿/存款池） | 真实资产账户（§11） |
| --- | --- | --- |
| 本质 | 记账激励与管控用的**虚拟额度** | 用户自记的**渠道余额** |
| 与限额 | 结余池直接影响当日可用额度 | 日常账户消费计入日限额 |
| 互相同步 | **不自动**与微信/银行卡余额同步 | 独立维护 |
| 用户提示 | 全产品显著标注「虚拟记账，非真实账户」 | 标注「用户自记，仅供参考」 |

发薪等**收入**进入资产账户（V1.5+），**不会**自动增加虚拟结余池或心愿，避免「账面突然多出可花额度」。

### 8.3 收入记账规则

1. **不计入日消费限额**：日限额仅统计**支出**。
2. **计入账本统计**：收入按所选账本/分类归集，参与月度收支报表。
3. **资产账户（V1.5+）**：可选关联收款账户，增加对应账户余额。
4. **常见类型**：工资、兼职、红包、退款、理财分红（分红进投资账户或日常账户，由用户选）。
5. **与虚拟池关系**：收入不自动进入结余池/心愿/存款池；用户若要把「发薪」视为虚拟攒钱，需手动操作「虚拟存入心愿」或「调整结余池」（V1.5 前仅支持心愿手动存入）。
6. **退款**：关联原支出账目时，可部分/全额冲减原分类统计；是否恢复当日限额由「退款是否计入当日消费抵扣」开关决定，**默认退款恢复**对应日的可用额度。

### 8.4 囤货贴纸：首次采购记账

采用 **「采购记总账 + 消耗记分摊」** 模式（推荐默认）：

1. **创建贴纸并勾选「同步记采购支出」**：生成一笔支出 = 单价 × 初始库存，计入绑定分类、账本、日限额；贴纸库存建立。
2. **消耗时**：再记一笔分摊支出 = 单价 × 消耗数量，用于反映**当日真实消耗**；同时在内部标记与采购总账的关联，**统计报表去重**时以消耗为准、采购可标记为「已分摊」避免双计（实现层：采购记录打 `stock_purchase` 标签，消耗打 `stock_consume`，月报表按 consume 汇总）。
3. **仅建贴纸不记采购**（高级选项）：适合「历史囤货补录」；之后每次消耗正常记支出。

### 8.5 日切与时间边界

1. **日切时刻**：按用户手机**本地时区**每日 24:00；云函数定时任务以用户 `timezone` 字段分批触发，或使用「打开 App 时检查是否跨日」作为补偿。
2. **补录历史**：用户可选账目日期；限额、挑战按**账目所属自然日**归属，非录入时刻。
3. **跨日编辑/删除**：修改后重算该日及受影响的结余池、挑战状态；若该日已分配结余且后续日已消费，回滚时提示「可能影响后续可用额度」。
4. **凌晨消费**：以用户选择的消费时间为准；默认当前时间。

### 8.6 新用户引导（Onboarding）

**V1.0 首次打开流程（4 步，可跳过但建议完成）**：

1. **欢迎页**：一句话说明「日限额 + 结余攒钱」核心价值；标注虚拟记账免责声明。
2. **设日限额**：输入日基准限额（提供「月薪 ÷ 30」计算器参考）。
3. **记第一笔账**：引导完成一笔支出演示（可跳过真实保存）。
4. **建第一个心愿**（可选）：输入名称与目标金额；说明结余可转入心愿。

完成后进入首页；未完成者在首页顶部展示「继续设置」轻提示。第 2 步完成后请求订阅消息授权。

### 8.7 游客模式与云端同步

1. **游客数据**：仅存本地 `uni.storage`；功能与登录用户一致（V1.0 范围）。
2. **登录合并策略**：
   - 云端无数据 → 本地全量上传；
   - 本地无数据 → 拉取云端；
   - **双方都有数据** → 弹窗让用户选择「以云端为准 / 以本地为准 / 合并（按记录时间戳去重，冲突条目人工确认）」。
3. **合并后**清除游客标识，openid 作为唯一主键。

### 8.8 分类管理规则

**预置支出分类**：餐饮、交通、购物、居住、娱乐、医疗、教育、人情、其他。

**预置收入分类**：工资、兼职、理财收益、红包、退款、其他。

规则：预置分类可隐藏不可删；自定义分类上限 20 个；删除时合并至目标分类；图标支持 emoji 或内置图标库。

### 8.9 核心数据模型（概要）

| 集合/表 | 核心字段 | 说明 |
| --- | --- | --- |
| `users` | openid, timezone, settings, onboarding_done, jar_skin_id, jar_skins_unlocked[] | 用户与全局设置；皮肤见 §12.6 |
| `transactions` | type, amount, category_id, ledger_id, account_id, date, stickers | 收支流水 |
| `ledgers` | name, icon, is_default | 场景账本 |

| `daily_limits` | date, base_limit, consumed, surplus, status | 日限额快照 |
| `surplus_pool` | balance, logs[] | 累计结余池 |

## 优化建议与补充提示

针对你原始想法中的逻辑漏洞与可优化点，补充如下：

1. **虚拟资金边界需明确**：所有「存款、心愿、惩罚扣除」均为记账层面的虚拟额度，不涉及真实资金流转。产品内必须显著提示，避免用户误解为金融账户，同时规避合规风险。
2. **超额惩罚须虚拟化（已与 §8 统一）**：P2 超额惩罚仅采用虚拟机制（扣连续天数、熄徽章、扣虚拟积分），不从存款池扣减记账额度。
3. **结余滚存逻辑（详见 §8.1）**：「当日基准限额 + 累计结余池」双池结构；消费先扣基准再扣结余；日结可分配至滚存/心愿/存款池。
4. **贴纸与分类绑定**：创建囤货贴纸时必须绑定对应支出分类，确保消耗记账时自动归入正确分类，保证统计数据准确；普通贴纸可选绑定默认分类，从素材库选用记账时自动带出，用户仍可手动修改。
5. **挑战与账本的关联规则**：建议日挑战默认统计总账本，支持用户自定义绑定单个子账本（如只挑战餐饮账本），适配更细分的管控场景。
6. **数据迁移能力预留**：云开发阶段需预留数据导出接口，后续升级 App 时支持用户无缝迁移记账数据，避免流失。
7. **餐次热量模块宜后置、宜可选**：与核心「限额 + 结余 + 攒钱」无强依赖，建议 V2.0 以可选模块上线；热量数据标注为用户自填/估算，AI 建议仅作 lifestyle 参考并加免责声明，避免健康类合规风险。
8. **热量缺口与 BMR/TDEE**：「还可摄入 = 目标 − 已摄入」满足大多数人；减脂模式下用**公式自动测算 BMR/TDEE**（身高体重年龄等），AI 只辅助解读活动强度、不直接产数值；BMR / 系数 / TDEE / 当日运动**全程可手动改**，兼顾省心与可控。
9. **账本与资产账户勿混**：账本=场景分类，资产账户=资金渠道；**旅行基金**等账本名≠投资理财；懒人可直接改账户余额，**可支配总余额**仅汇总日常账户；公积金（专项）、基金/股票/理财（投资）与日常零钱默认均不计入日限额。
10. **MVP 边界以 §七 V1.0 为准**：V1.0 不含存款池、贴纸、资产账户、OCR；开发排期（§六）与版本规划保持一致，避免首版 scope 膨胀。
11. **导航与配色**：底部采用一体化悬浮胶囊 + 中间凸出「记一笔」；品牌主色为绿 `#00BF4B`，「我的」走顶栏头像。
